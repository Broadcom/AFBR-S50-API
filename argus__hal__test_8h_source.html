70 /*!***************************************************************************
71  * @brief Executes a series of tests in order to verify the HAL implementation.
72  *
73  * @details A series of automated tests are executed on the target platform in
74  * order to verify the implementation of the HAL that are required by
75  * the API.
76  *
77  * Each test will write an error description via the print (i.e. UART)
78  * function that shows what went wrong. Also an corresponding status is
79  * returned in case no print functionality is available.
80  *
81  * The following tests are executed:
82  *
83  * **1) Timer Plausibility Test:**
84  *
85  * Rudimentary tests of the lifetime counter (LTC) implementation.
86  * This verifies that the LTC is running by checking if the returned
87  * values of two consecutive calls to the #Timer_GetCounterValue
88  * function are ascending. An artificial delay using the NOP operation
89  * is induced such that the timer is not read to fast.
90  *
91  * **2) Timer Wraparound Test:**
92  *
93  * The LTC values must wrap from 999999 µs to 0 µs and increase the
94  * seconds counter accordingly. This test verifies the correct wrapping
95  * by consecutively calling the #Timer_GetCounterValue function until
96  * at least 2 wraparound events have been occurred.
97  *
98  * **3) SPI Connection Test:**
99  *
100  * This test verifies the basic functionality of the SPI interface.
101  * The test utilizes the devices laser pattern register, which can
102  * be freely programmed by any 128-bit pattern. Thus, it writes a byte
103  * sequence and reads back the written values on the consecutive SPI
104  * access.
105  *
106  * **4) SPI Interrupt Test:**
107  *
108  * This test verifies the correct implementation of the device
109  * integration finished interrupt callback. Therefore it configures
110  * the device with a minimal setup to run a pseudo measurement that
111  * does not emit any laser light.
112  *
113  * Note that this test does verify the GPIO interrupt that occurs
114  * whenever the device has finished the integration/measurement and
115  * new data is waiting to be read from the device. This does not test
116  * the interrupt that is triggered when the SPI transfer has finished.
117  *
118  * The data ready interrupt implies two S2PI layer functions that
119  * are tested in this test: The #S2PI_SetIrqCallback function installs
120  * a callback function that is invoked whenever the IRQ occurs.
121  * The IRQ can be delayed due to higher priority task, e.g. from the
122  * user code. It is essential for the laser safety timeout algorithm
123  * to determine the device ready signal as fast as possible, another
124  * method is implemented to read if the IRQ is pending but the
125  * callback has not been reset yet. This is what the #S2PI_ReadIrqPin
126  * function is for.
127  *
128  * **5) GPIO Mode Test:**
129  *
130  * This test verifies the GPIO mode of the S2PI HAL module. This is
131  * done by leveraging the EEPROM readout sequence that accesses the
132  * devices EEPROM via a software protocol that depends on the GPIO
133  * mode.
134  *
135  * This the requires several steps, most of them are already verified
136  * in previous tests:
137  *
138  * - Basic device configuration and enable EEPROM.
139  * - Read EERPOM via GPIO mode and apply Hamming weight.
140  * - Repeat several times (to eliminate random readout issues).
141  * - Decode the EEPROM (using EEPROM_Decode in argus_cal_eeprom.c).
142  * - Check if Module Number and Chip ID is not 0.
143  *
144  * **6) Timer Test for Lifetime Counter:**
145  *
146  * The test verifies the lifetime counter timer HAL implementation by
147  * comparing the timings to the AFBR-S50 device as a reference.
148  * Therefore several measurement are executed on the device, each with
149  * a different averaging sample count. The elapsed time increases
150  * linearly with the number of averaging samples. In order to remove
151  * the time for software/setup, a linear regression fit is applied to
152  * the measurement results and only the slope is considered for the
153  * result. A delta of 102.4 microseconds per sample is expected.
154  * If the measured delta per sample is within an specified error range,
155  * the timer implementation is considered correct.
156  *
157  * **7) Timer Test for Periodic Interrupt Timer:**
158  *
159  * The test verifies the correct implementation of the periodic
160  * interrupt timer (PIT). It sets different intervals and waits for
161  * a certain number of interrupts to happen. Each interrupt event
162  * is counted and the time between the first and the last interrupt
163  * is measured. Finally, the measured interval is compared to the
164  * expectations.
165  *
166  *
167  * @param spi_slave The SPI hardware slave, i.e. the specified CS and IRQ
168  * lines. This is actually just a number that is passed
169  * to the SPI interface to distinct for multiple SPI slave
170  * devices. Note that the slave must be not equal to 0,
171  * since is reserved for error handling.
172  *
173  * @return Returns the \link #status_t status\endlink (#STATUS_OK on success).
174  *****************************************************************************/
175 status_t Argus_VerifyHALImplementation(s2pi_slave_t spi_slave);
