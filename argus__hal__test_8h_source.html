71/*!***************************************************************************
72 * @brief Executes a series of tests in order to verify the HAL implementation.
73 *
74 * @details A series of automated tests are executed on the target platform in
75 * order to verify the implementation of the HAL that are required by
76 * the API.
77 *
78 * Each test will write an error description via the print (i.e. UART)
79 * function that shows what went wrong. Also an corresponding status is
80 * returned in case no print functionality is available.
81 *
82 * The following tests are executed:
83 *
84 * **1) Timer Plausibility Test:**
85 *
86 * Rudimentary tests of the lifetime counter (LTC) implementation.
87 * This verifies that the LTC is running by checking if the returned
88 * values of two consecutive calls to the #Timer_GetCounterValue
89 * function are ascending. An artificial delay using the NOP operation
90 * is induced such that the timer is not read to fast.
91 *
92 * **2) Timer Wraparound Test:**
93 *
94 * The LTC values must wrap from 999999 µs to 0 µs and increase the
95 * seconds counter accordingly. This test verifies the correct wrapping
96 * by consecutively calling the #Timer_GetCounterValue function until
97 * at least 2 wraparound events have been occurred.
98 *
99 * **3) SPI Connection Test:**
100 *
101 * This test verifies the basic functionality of the SPI interface.
102 * The test utilizes the devices laser pattern register, which can
103 * be freely programmed by any 128-bit pattern. Thus, it writes a byte
104 * sequence and reads back the written values on the consecutive SPI
105 * access.
106 *
107 * **4) SPI Interrupt Test:**
108 *
109 * This test verifies the correct implementation of the device
110 * integration finished interrupt callback. Therefore it configures
111 * the device with a minimal setup to run a pseudo measurement that
112 * does not emit any laser light.
113 *
114 * Note that this test does verify the GPIO interrupt that occurs
115 * whenever the device has finished the integration/measurement and
116 * new data is waiting to be read from the device. This does not test
117 * the interrupt that is triggered when the SPI transfer has finished.
118 *
119 * The data ready interrupt implies two S2PI layer functions that
120 * are tested in this test: The #S2PI_SetIrqCallback function installs
121 * a callback function that is invoked whenever the IRQ occurs.
122 * The IRQ can be delayed due to higher priority task, e.g. from the
123 * user code. It is essential for the laser safety timeout algorithm
124 * to determine the device ready signal as fast as possible, another
125 * method is implemented to read if the IRQ is pending but the
126 * callback has not been reset yet. This is what the #S2PI_ReadIrqPin
127 * function is for.
128 *
129 * **5) GPIO Mode Test:**
130 *
131 * This test verifies the GPIO mode of the S2PI HAL module. This is
132 * done by leveraging the EEPROM readout sequence that accesses the
133 * devices EEPROM via a software protocol that depends on the GPIO
134 * mode.
135 *
136 * This the requires several steps, most of them are already verified
137 * in previous tests:
138 *
139 * - Basic device configuration and enable EEPROM.
140 * - Read EERPOM via GPIO mode and apply Hamming weight.
141 * - Repeat several times (to eliminate random readout issues).
142 * - Decode the EEPROM (using EEPROM_Decode in argus_cal_eeprom.c).
143 * - Check if Module Number and Chip ID is not 0.
144 *
145 * **6) Timer Test for Lifetime Counter:**
146 *
147 * The test verifies the lifetime counter timer HAL implementation by
148 * comparing the timings to the AFBR-S50 device as a reference.
149 * Therefore several measurement are executed on the device, each with
150 * a different averaging sample count. The elapsed time increases
151 * linearly with the number of averaging samples. In order to remove
152 * the time for software/setup, a linear regression fit is applied to
153 * the measurement results and only the slope is considered for the
154 * result. A delta of 102.4 microseconds per sample is expected.
155 * If the measured delta per sample is within an specified error range,
156 * the timer implementation is considered correct.
157 *
158 * **7) Timer Test for Periodic Interrupt Timer:**
159 *
160 * The test verifies the correct implementation of the periodic
161 * interrupt timer (PIT). It sets different intervals and waits for
162 * a certain number of interrupts to happen. Each interrupt event
163 * is counted and the time between the first and the last interrupt
164 * is measured. Finally, the measured interval is compared to the
165 * expectations.
166 *
167 *
168 * @param spi_slave The SPI hardware slave, i.e. the specified CS and IRQ
169 * lines. This is actually just a number that is passed
170 * to the SPI interface to distinct for multiple SPI slave
171 * devices. Note that the slave must be not equal to 0,
172 * since is reserved for error handling.
173 *
174 * @return Returns the \link #status_t status\endlink (#STATUS_OK on success).
175 *****************************************************************************/
176status_t Argus_VerifyHALImplementation(s2pi_slave_t spi_slave);
