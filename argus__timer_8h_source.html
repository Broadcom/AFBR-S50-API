40 /*!***************************************************************************
41  * @defgroup argus_timer Timer: Hardware Timer Interface
42  * @ingroup argus_platform
43  *
44  * @brief Timer implementations for lifetime counting as well as periodic
45  * callback.
46  *
47  * @details The module provides an interface to the timing utilities that
48  * are required by the AFBR-S50 time-of-flight sensor API.
49  *
50  * Two essential features have to be provided by the user code:
51  * 1. Time Measurement Capability: In order to keep track of outgoing
52  * signals, the API needs to measure elapsed time. In order to
53  * provide optimum device performance, the granularity should be
54  * around 10 to 100 microseconds.
55  * 2. Periodic Callback: The API provides an automatic starting of
56  * measurement cycles at a fixed frame rate via a periodic
57  * interrupt timer. If this feature is not used, implementation
58  * of the periodic interrupts can be skipped. An weak default
59  * implementation is provide in the API.
60  * .
61  *
62  * The time measurement feature is simply implemented by the function
63  * #Timer_GetCounterValue. Whenever the function is called, the
64  * provided counter values must be written with the values obtained
65  * by the current time.
66  *
67  * The periodic interrupt timer is a simple callback interface.
68  * After installing the callback function pointer via #Timer_SetCallback,
69  * the timer can be started by setting interval via #Timer_SetInterval.
70  * From then, the callback is invoked periodically as the corresponding
71  * interval may specify. The timer is stopped by setting the interval
72  * to 0 using the #Timer_SetInterval function. The interval can be
73  * updated at any time by updating the interval via the #Timer_SetInterval
74  * function. To any of these functions, an abstract parameter pointer
75  * must be passed. This parameter is passed back to the callback any
76  * time it is invoked.
77  *
78  * In order to provide the usage of multiple devices, an mechanism is
79  * introduced to allow the installation of multiple callback interval
80  * at the same time. Therefore, the abstract parameter pointer is used
81  * to identify the corresponding callback interval. For example, there
82  * are two callbacks for two intervals, t1 and t2, required. The user
83  * can start two timers by calling the #Timer_SetInterval method twice,
84  * but with an individual parameter pointer, ptr1 and ptr2, each:
85  * \code
86  * Timer_SetInterval(100000, ptr1); // 10 ms callback w/ parameter ptr1
87  * Timer_SetInterval(200000, ptr2); // 20 ms callback w/ parameter ptr1
88  * \endcode
89  *
90  * Note that the implemented timer module must therefore support
91  * as many different intervals as instances of the AFBR-S50 device are
92  * used.
93  *
94  * @addtogroup argus_timer
95  * @{

104 /*!***************************************************************************
105  * @brief Obtains the lifetime counter value from the timers.
106  *
107  * @details The function is required to get the current time relative to any
108  * point in time, e.g. the startup time. The returned values \p hct and
109  * \p lct are given in seconds and microseconds respectively. The current
110  * elapsed time since the reference time is then calculated from:
111  *
112  * t_now [µsec] = hct * 1000000 µsec + lct * 1 µsec
113  *
114  * Note that the accuracy/granularity of the lifetime counter does
115  * not need to be 1 µsec. Usually, a granularity of approximately
116  * 100 µsec is sufficient. However, in case of very high frame rates
117  * (above 1000 frames per second), it is recommended to implement
118  * an even lower granularity (somewhere in the 10 µsec regime).
119  *
120  * It must be guaranteed, that each call of the #Timer_GetCounterValue
121  * function must provide a value that is greater or equal, but never lower,
122  * than the value returned from the previous call.
123  *
124  * A hardware based implementation of the lifetime counter functionality
125  * would be to chain two distinct timers such that counter 2 increases
126  * its value when counter 1 wraps to 0. The easiest way is to setup
127  * counter 1 to wrap exactly every second. Counter 1 would than count
128  * the sub-seconds (i.e. µsec) value (\p lct) and counter 2 the seconds
129  * (\p hct) value. A 16-bit counter is sufficient in case of counter 1
130  * while counter 2 must be a 32-bit version.
131  *
132  * In case of a lack of available hardware timers, a software solution
133  * can be used that requires only a 16-bit timer. In a simple scenario,
134  * the timer is configured to wrap around every second and increase
135  * a software counter value in its interrupt service routine (triggered
136  * with the wrap around event) every time the wrap around occurs.
137  *
138  *
139  * @note The implementation of this function is mandatory for the correct
140  * execution of the API.
141  *
142  * @param hct A pointer to the high counter value bits representing current
143  * time in seconds.
144  *
145  * @param lct A pointer to the low counter value bits representing current
146  * time in microseconds. Range: 0, .., 999999 µsec
147  *****************************************************************************/
148 void Timer_GetCounterValue(uint32_t * hct, uint32_t * lct);
